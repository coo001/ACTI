## 1. 제품 개요

### 1.1 한 줄 정의
> **연극영화과 학생을 위한, 4축 16유형 위트 진단 + 캡처 공유 최적화 결과지로 단톡방·SNS에서 깔깔거릴 수 있는 연기 정체성 놀이거리.**

### 1.2 배경
연영과 학생의 일상은 "연기"로 가득한데, 정작 그 연기 자체를 가볍게 놀 거리는 시장에 비어 있다. 진지한 연기론은 있지만, **단톡방에서 같이 돌리고 결과 캡처 던지며 깔깔거릴 콘텐츠**가 없다. 이걸 채운다.

### 1.3 제품 가치 (우선순위)
| 순위 | 가치 | 설명 |
|---|---|---|
| 1차 | **공유·놀이** | 동기·동아리 친구와 함께 풀고 비교하는 재미 |
| 2차 | **정체성 표현** | "나 이런 배우야"를 한 줄로 던질 수 있는 문구 |
| 3차 | **희미한 자기 통찰** | 결과지 "어울리는 역할/장르"로 곱씹어볼 거리 |

### 1.4 5대 설계 원칙
1. 공유 가능성 > 분석 정확도
2. 캐릭터성 강한 위트 네이밍
3. 짧은 테스트 (12~14문항, 2~3분)
4. 결과지 비교 재미 1초 인지
5. 일반 진단으로 환원되지 않는 도메인 양념

## 3. 목표 (Stage 1 범위)

### 3.1 제품 목표 (v1)
- **콘텐츠 완성**: 12~14 시나리오 문항 + 16유형 콘텐츠 풀세트
- **공유 마찰 제거**: 이미지 저장 / URL 공유 / 카톡 공유 모두 1탭
- **모바일 first**: 메인 사용 환경에서 가로 스크롤·확대 없이 풀림
- **β 테스트 검증**: 학과 친구 10명 풀이로 유형 분포 검증

### 3.2 명시적 비목표 (Won't)
- 영어/다국어
- 256셀(16×16) 궁합도 매트릭스
- 명대사 추천 (Stage 2 후보)
- 사용자 계정/로그인

## 6. 사용자 흐름 요약

(전문은 `user-flow.md`)

### 6.1 메인 흐름
```
[유입 채널] → ① 랜딩 → ② 문항 풀이 (12~14) → [점수 계산] → ③ 결과 페이지
```

### 6.2 핵심 분기 3개
- **분기 1 공유** — 이미지 저장 / URL 복사 / 카톡 공유 → 외부 채널 → 수신자 진입 (③')
- **분기 2 재진입** — "다시 풀기" → ② 1번 문항
- **분기 3 사슬 연장** — 천적/베프 카드 → ③″ (다른 유형 결과지) → 사슬 무한

### 6.3 핵심 확산 루프
```
유저 A 풀기 → 공유 → 유저 B 수신 → "나도 풀어보기" → 유저 B 풀기 → 공유 → 유저 C ...
```

→ 모든 화면 설계는 이 루프의 마찰을 줄이는 방향.

### 6.4 흐름 설계 결정 (Stage 1.3)
| 결정 항목 | 선택 |
|---|---|
| 문항 풀이 중 뒤로가기 | ✅ 있음 (이전 답 보존) |
| 문항 진행 | 선택 즉시 자동 다음 (탭 1회) |
| 공유 URL 수신자 진입 | 결과지 바로 표시 + "나도 풀어보기" CTA 강조 |

## 8. 콘텐츠 명세

### 8.1 4축 정의

| # | 축 | 양극 | 약자 |
|---|---|---|---|
| 1 | 접근 철학 | 메소드 ↔ 테크닉 | M / T |
| 2 | 현장 운영 | 즉흥 ↔ 설계 | I / P |
| 3 | 준비 사고법 | 직관 ↔ 분석 | N / A |
| 4 | 표현 통로 | 신체 ↔ 내면 | B / S |

→ 4글자 조합으로 16유형 (MINB, TPAS, MIAS, TINB …)

### 8.2 16유형 콘텐츠 분량 (작성 대상)

| 항목 | 분량 | 비고 |
|---|---|---|
| 위트 네이밍 | 16개 (각 1줄) | 톤 일관성 5점 자체 평가 |
| 한 줄 캐치프레이즈 | 16개 | SNS 캡션용 |
| 캐릭터 묘사 | 16개 × 3~4줄 | 위트+공감 |
| 어울리는 역할/장르 | 16개 × 3~4개 | 차별점 |
| 천적 유형 매핑 | 16 → 1 | 16세트 |
| 베프 유형 매핑 | 16 → 1 | 16세트 |

### 8.3 문항 분량
- **시나리오**: 12~14개
- **소재 풀**: 리허설, 대본 첫 받음, 공연 직전, 사후 평가, 캐스팅 거절, 슬럼프, 합 안 맞음 등
- **구조**: 1 문항 = 4 선택지 = 4축 중 한 축의 4극(각 극 1개씩) — 또는 한 문항당 한 축의 양극 + 다른 축 영향(설계 단계 결정)
- **워크플로우**: AI 초안 → 본인 폴리싱

### 8.4 톤 가이드
- 진지 30 / 재미 70
- 시적·풍자적·B급
- 단톡방 짤맛 최적화
- 일반 진단으로 환원 금지 (도메인 양념 필수)

## 9. 비기능 요구사항

### 9.1 성능
| 항목 | 목표 |
|---|---|
| 첫 의미있는 콘텐츠 | ≤ 2초 (3G 빠름 기준) |
| 정적 호스팅 | Vercel/Netlify 호환 |
| 번들 크기 | ≤ 200KB gzip (이상적) |

### 9.2 호환성 (실기 검증 대상)
- iOS Safari (최신 2버전)
- Android Chrome (최신 2버전)
- 데스크톱 Chrome (보조)

### 9.3 접근성
- WCAG AA 대비 (텍스트 4.5:1)
- 터치 타겟 ≥ 44×44px
- 폰트 최소 본문 14~16px

### 9.4 데이터·상태
- 사용자 계정 없음
- 답안: 메모리 (세션)
- 본인 결과 코드: localStorage (영구)
- 16유형 콘텐츠: 정적 JSON/TS 모듈 (빌드 시점 고정)

### 9.5 SEO/공유
- OG 태그 16유형별 동적 (제목/설명/이미지)
- 카카오톡 공유 미리보기 카드 16개

## 11. 성공 지표

### 11.1 정성 (v1 출시 기준)
- 학과 친구 10명 β 테스트에서 "캡처해서 단톡방에 올리고 싶다" 응답 ≥ 7명
- 톤 일관성 자체 평가 5점 척도에서 16유형 모두 평균 ≥ 3.5점

### 11.2 정량 (출시 후 2주)
- 유형 분포가 한쪽으로 70% 이상 쏠리지 않음
- 결과 페이지 도달률(랜딩→결과) ≥ 60%
- 공유 액션 클릭률(결과 도달자 중) ≥ 30%
- (β 통계 인프라가 있다면) URL 공유 수신 진입 ≥ 자기 풀이 진입의 0.3배
