## 2. 선택 근거 (Why Web)

### 2.1 PRD 정합성
- **PRD 6.4 / 9.2 / 9.4**: 정적 호스팅 (Vercel/Netlify), 카톡 OG 카드, URL 인코딩 결과 페이지(`/result/{코드}`)
- 모두 웹에서 자연스럽게 구현되는 항목

### 2.2 확산 메커니즘 정합성
- **PRD 6.3 확산 루프**: "공유 → 수신 → 본인 풀이 → 다시 공유"
- 앱은 설치 마찰 때문에 이 루프가 끊김 (스토어 우회 → 앱 설치 → 진입)
- 웹은 **링크 한 번 클릭 = 즉시 결과** → 루프 마찰 0

### 2.3 사이드 프로젝트 가치관 일치
- idea-brief 11항: "빠른 출시·완성도 최적화 프로젝트가 아니라, **콘텐츠 퀄리티에 시간 투입**"
- 네이티브 앱 빌드·배포 사이클은 콘텐츠 시간을 잡아먹음

### 2.4 메인 페르소나 환경
- 연영과 학생 일상 = 단톡방/에타/인스타
- 모두 **링크 기반 공유**가 1차 채널 → 웹이 가장 빠른 경로

## 3. 모바일 first 반응형 웹의 구체적 의미

| 항목 | 결정 |
|---|---|
| 1차 설계 기준 폭 | **360px ~ 430px** (모바일 세로) |
| 2차 (보조) 폭 | 480px 컨테이너 최대 — 데스크톱에서도 모바일 폭 유지 |
| 터치 first | 모든 버튼 ≥ 44×44px (Apple HIG) |
| 호버 의존 금지 | 데스크톱 호버 효과는 보조 장식, 핵심 정보 전달 금지 |
| 가로 모드 | 디자인 우선순위 낮음 (대응은 하되 최적화 안 함) |
| PWA 여부 | v1 범위 밖 (Could 후보) |

## 4. 디자인 작업 시 유의 사항

### 4.1 정적 SPA 전제
- 빌드 산출물이 정적 파일이어야 함 (서버 API 호출 없음)
- 16유형 콘텐츠는 빌드 시점 고정 (정적 JSON/TS 모듈)
- 동적 OG 태그는 빌드 시점에 16유형용 16개 페이지 사전 생성 (Next.js static export 등)

### 4.2 모바일 first 디자인 원칙
- **Vertical first**: 세로 스크롤 외 인터랙션 최소화
- **One-thumb reachable**: 주요 CTA가 화면 하단 1/3에 위치
- **No hover dependency**: 데스크톱은 모바일 UI 그대로 노출 (별도 최적화 안 함)

### 4.3 공유 채널 최적화 (Web 특화)
- OG 태그 / Twitter 카드 16유형 동적 (디자인 산출물에 OG 이미지 템플릿 포함)
- 카카오톡 공유 SDK 호환 미리보기 카드 디자인 (정사각형 800×800 권장)
- 결과지 이미지 캡처 = html2canvas → 캡처 영역 디자인은 PNG 추출 친화적으로 (그라데이션·복잡한 효과 주의)

## 6. Step 2.1로 넘기는 것

`screen-structure.md`에 정의된 5개 화면(S1·S2·S3·S3'·S4)을 **목적 / 유저 상황 / 정보 위계 / 핵심 액션** 4축으로 분석:

1. 각 화면이 "왜 이 시점에 존재하는가" (목적)
2. 사용자가 어떤 감정·맥락·디바이스 환경에서 도달하는가 (유저 상황)
3. 화면 안에서 무엇이 가장 중요·덜 중요한가 (정보 위계)
4. 사용자가 이 화면에서 반드시 해야 하는 1~2가지 행동 (핵심 액션)
