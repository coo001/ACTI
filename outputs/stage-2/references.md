# References — 연기 스타일 MBTI

> Stage 2.2 산출물 — 레퍼런스 분석 + 비주얼 방향성 도출
> 작성일: 2026-05-14

## 비주얼 방향성 결정 (Stage 2.2)

**MBTI 커뮤니티 친수적 파스텔 (16Personalities 스타일)**

→ 단, idea-brief의 "**진지 30 / 재미 70 / 위트 B급 / 짤맛**" 톤이 죽지 않게 **파스텔 + 위트 카피 결합** 으로 방향 보정.

### 비주얼 방향성 슬로건

> **"파스텔로 입고, 카피로 비튼다."**

- **시각 시스템**: 파스텔 백그라운드 + 둥근 카드 + 친근한 일러스트 (16Personalities 톤)
- **카피·콘텐츠 시스템**: 위트·풍자·B급 (idea-brief 톤)
- **두 시스템의 충돌 처리**: 시각은 친수성 우선, 카피는 도메인 위트 우선
- **이유**: 친근한 시각 = 진입 마찰 ↓, 위트 카피 = 단톡방 공유 동력 ↑

## 1. 레퍼런스 분석 (화면별)

### S1 — 랜딩 페이지

#### 1-A. 16Personalities (`16personalities.com`)
**왜 보는가**: MBTI 진단 랜딩의 표준. 친수성 베이스라인.

**가져올 점**
- 첫 뷰포트 안에 명확한 시작 CTA (큰 primary 버튼)
- "Free", "약 12분" 식의 분량 표시가 CTA 근처에 작게
- 캐릭터 일러스트가 시선 끌기용으로 우상단
- 톤: 무난·친근

**버릴 점**
- 톤이 너무 무난 (위트 0%) → idea-brief 정체성 죽음
- 분량이 너무 김 (12분) → 우리는 2~3분
- 16개 캐릭터 그리드 미리보기 → 우리는 첫 화면에 안 노출 (호기심 보존)

**디자인 시그널 추출**
- 첫 뷰포트 안 CTA + 분량 표시 = 표준 패턴 채택
- 일러스트는 위트 한 컷 (B급 톤)으로 변형

---

#### 1-B. Buzzfeed 퀴즈 랜딩 (`buzzfeed.com/quiz`)
**왜 보는가**: 위트·B급 진단의 표준.

**가져올 점**
- 도메인이 특화된 한 줄 헤드라인 ("Only [specific] Will Get This Quiz")
- 일러스트보다 짤·이모지 위주의 시각 (요리 잘 통하는 톤)
- "Let's Start" CTA가 굵고 친근한 폰트

**버릴 점**
- 광고 위주 레이아웃 → 우리는 광고 0
- 양산형 톤 → 우리는 연영과 도메인 깊이로 차별
- 이모지 다용 → 우리는 이모지 금지(SKILL.md), 아이콘으로 대체

**디자인 시그널 추출**
- 도메인 특화 한 줄 헤드라인 = 채택 ("연영과 너만 이해할 결과")
- 진지/광고 무드 0 = 채택

### S2 — 문항 페이지

#### 2-A. 16Personalities 문항 페이지
**왜 보는가**: 단일 문항 카드 + 진행률 + 자동 진행의 표준.

**가져올 점**
- 상단 진행바 + "n / m" 카운터
- 1문항 = 1뷰포트 (스크롤 없이 풀림)
- 선택지 클릭 = 즉시 다음 (자동 진행)
- 동의/반대형 양극 슬라이더 (1~7) — 단, 우리는 4지선다라 그대로 안 씀

**버릴 점**
- 7점 슬라이더 → 우리는 4지선다 카드
- "다음" 버튼 명시적 표시 → 우리는 선택 = 자동 진행 (마찰 더 낮춤)
- 무미건조한 톤 → 위트로 변형

**디자인 시그널 추출**
- 진행바 상단 고정 = 채택
- 1문항 1뷰포트 = 채택
- 자동 진행 = 채택 (16P보다 한 단계 더 마찰 낮춤)

---

#### 2-B. Akinator (`akinator.com`)
**왜 보는가**: 1문항 1뷰포트 + 4지선다 + 시각적 캐릭터 동행.

**가져올 점**
- 4개 선택지가 카드 형태로 동일 크기 (2x2 그리드 또는 세로 4개)
- 캐릭터 일러스트가 화면 한쪽에서 사용자와 "대화"하는 느낌
- 선택 시 즉각적 시각 피드백 (카드 강조 + 0.2~0.3초 후 진행)

**버릴 점**
- 게임 UI 톤 → 우리는 진단형으로 정돈
- 뒤로가기 버튼이 모호 → 우리는 명확한 ← 버튼

**디자인 시그널 추출**
- 4지선다 카드 시각 균형 = 채택
- 선택 시 시각 피드백 0.2~0.3초 = 채택

---

#### 2-C. Persona 진단형 마이크로사이트 (예: Spotify "Find Your Sound")
**왜 보는가**: 시나리오형 도메인 깊이 문항 + 콘텐츠 자체가 위트.

**가져올 점**
- 시나리오 카피가 1~3줄로 풍부하지만 정보 과부하 아님
- 선택지 카피가 캐릭터화 ("A는 ~한다" 식 묘사형)
- 카드에 약간의 일러스트·아이콘 (글자만 있는 4지선다 지루함 회피)

**디자인 시그널 추출**
- 시나리오 카피 1~3줄 = 채택
- 선택지에 도메인 아이콘 1개씩 (대본/리허설/캐스팅 등) = 채택 검토

### S3 — 결과 페이지 (캡처 + 스크롤)

#### 3-A. 16Personalities 결과 페이지
**왜 보는가**: 16유형 결과지의 표준 구조.

**가져올 점**
- 유형 코드 (예: INFP) 큰 라벨 + 유형 별명("Mediator") 함께 표시
- 캐릭터 일러스트가 큼 (정체성 시각화)
- 스크롤 영역: 강점·약점·관계·진로 등 깊이 있는 콘텐츠
- 색상 시스템: 4그룹(분석가/외교관/관리자/탐험가)별 컬러 코드

**버릴 점**
- 결과 페이지 본문이 5000자+ → 우리는 모바일 캡처용으로 압축
- 본문이 "글" 위주 → 우리는 "캡처 카드" 위주
- 무난·중성 톤 → 우리는 위트 강조

**디자인 시그널 추출**
- 4글자 코드 + 별명 함께 표시 = 채택 (코드 `MINB` + 위트 별명 "비 오는 날 뛰쳐나가는 햄릿형")
- 유형 그룹별 컬러 시스템 = 채택 검토 (4그룹? 16개 각자? — Step 2.4에서 결정)
- 캐릭터 일러스트 큼 = 채택 (단, 캡처 영역 1뷰포트 안에 들어가는 크기로 조정)

---

#### 3-B. Spotify Wrapped 공유 카드
**왜 보는가**: 캡처/공유 친화 결과 화면의 정점.

**가져올 점**
- 단일 색 배경 + 큰 텍스트 + 한정된 시각 요소 → 캡처 시 임팩트 ★★★
- 유형별 색상 변주 (해마다 시그니처 컬러)
- 모바일 1뷰포트 한 장 = 한 카드 (스크롤로 카드 넘김)
- 워터마크가 하단에 작게 (Spotify 로고)
- "Share" 버튼이 1탭 액션 + 인스타 스토리 직접 연결

**버릴 점**
- 영상·애니메이션 의존 → 우리는 정적 캡처 우선
- 카드 슬라이드 형태 → 우리는 한 페이지 스크롤

**디자인 시그널 추출**
- 단일 색 배경 + 큰 텍스트 = 채택 (캡처 영역)
- 유형별 시그니처 컬러 = 채택
- 워터마크 = 채택 (캡처된 이미지에 서비스명 자동 포함)
- 1탭 공유 = 채택

---

#### 3-C. 한국 진단 사이트 (예: 8 Tones — 8tones.kr 류)
**왜 보는가**: 한국 단톡방 공유 문화에 최적화된 결과지.

**가져올 점**
- 위트 별명이 코드보다 크고 임팩트 (예: "혼자 다 해먹는 파스타 같은 사람")
- 캡처 영역이 모바일 1뷰포트에 압축 (코드+별명+묘사 3~4줄)
- 카카오톡 공유 버튼이 메인 액션 (한국 톡 문화)
- 친구 결과지 점프 ("당신의 천적: XX") = 사슬 연장 패턴

**디자인 시그널 추출**
- 위트 별명 = 시각 1순위 = 채택
- 1뷰포트 캡처 영역 압축 = 채택 (US-05 AC와 직결)
- 카톡 공유 메인 액션 = 채택
- 천적/베프 카드 = 채택 (이미 F12로 정의됨)

#### 3-D. 인스타 스토리 친화 결과지 (예: Tarot 진단류, 케이팝 멤버 매칭)
**왜 보는가**: 인스타 스토리 9:16 비율에 최적화된 결과 카드.

**가져올 점**
- 9:16 또는 4:5 세로 비율 카드
- 유형별 그라데이션 배경 (인스타 시각 매력)
- 결과 카드에 URL 또는 QR (확산 동력)

**디자인 시그널 추출**
- OG 이미지(인스타 스토리 공유용)는 1080×1350 (4:5) 또는 1080×1920 (9:16)으로 별도 생성
- 캡처 영역 디자인 자체는 모바일 뷰포트 친화 (3:4 정도), 인스타 공유는 별도 OG 이미지 템플릿

## 3. 비주얼 방향성 최종 명세

### 3.1 컬러 톤
- **베이스**: 따뜻한 파스텔 (크림·옅은 파우더 블루·복숭아 핑크·세이지 그린)
- **포인트**: 유형 그룹별 시그니처 컬러 (4그룹 분류 검토 — Step 2.4)
- **다크 모드**: v1 범위 밖 (Could)

### 3.2 형태
- **둥근 모서리** 큰 라디우스 (16~24px 카드, 12~16px 버튼)
- **그림자** 부드럽고 옅게 (멀티레이어 가능)
- **여백** 넉넉 (모바일에서도 답답하지 않게)

### 3.3 타이포그래피
- **헤드/네이밍**: 굵은 산세리프, 한글 가독성 우수 (예: Pretendard, Gmarket Sans)
- **본문**: 14~16px, 줄간격 1.6~1.7
- **코드 (`MINB`)**: 모노스페이스 또는 별도 강조 폰트 (배지 느낌)
- **위트 카피**: 본문 + 약간 더 큰 사이즈, 강조 컬러

### 3.4 일러스트/아이콘
- **캐릭터 일러스트**: v1은 SVG 도형 + 한두 가지 색 (단순한 아바타) — 16개 만들기 부담 ↓
  - 16유형 각자 일러스트 작성은 작업량 과부하 → **4그룹 × 일러스트 1개**로 시작 (Phase 2 이후 16개 확장 검토)
- **UI 아이콘**: Lucide (이모지 금지)
- **그래픽 장식**: 도형(원·곡선) 위주, 사진 0

### 3.5 인터랙션 톤
- **무게**: 가볍고 빠른 트랜지션 (150~250ms)
- **이징**: ease-out 중심 (자연스러운 감속)
- **마이크로인터랙션**: 선택 카드 살짝 들림 (translate Y -2px + 그림자 강화)
- **셀럽레이션**: 결과 페이지 진입 시 가벼운 fade-in + scale (과하지 않게)

### 3.6 UX 라이팅 톤
- **시각은 친수성, 카피는 위트** — 두 시스템 균형
- 결과지 네이밍 예: "비 오는 날 뛰쳐나가는 햄릿형" (위트) → 파스텔 카드 위에 굵게
- 안내 카피는 친근하지만 약간 능청 ("친구가 풀어본 결과예요. 너도 풀어볼래?")
- 광고·전문가 톤 일체 금지

## 6. Step 2.3로 넘기는 것

채택된 패턴들을 5개 화면(S1·S2·S3·S3'·S4)의 **ASCII/텍스트 와이어프레임**으로 변환:

- 정확한 요소 배치 (위/아래/좌/우)
- 컴포넌트 크기 비율
- 모바일 360~430px 폭 가정
- 캡처 영역의 1뷰포트 압축 검증
- 화면 간 일관 요소(헤더·진행바·CTA) 시각 통일

이 와이어프레임이 확정된 후에야 디자인 토큰(Step 2.4)으로 넘어감 — "구조 먼저, 톤 나중에" 원칙.
