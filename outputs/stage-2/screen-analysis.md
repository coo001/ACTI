# Screen Analysis — 연기 스타일 MBTI

> Stage 2.1 산출물 — 화면별 목적/유저상황/정보위계/핵심액션 분석
> 작성일: 2026-05-14

## 분석 프레임

각 화면은 다음 4축으로 분석:

| 축 | 정의 | 디자인 함의 |
|---|---|---|
| **목적** | 이 화면이 존재하는 단 하나의 이유 | 화면당 1순위 메시지 결정 |
| **유저 상황** | 이 화면에 도달할 때의 맥락·감정·환경 | 톤·여백·로딩 임계점 결정 |
| **정보 위계** | 중요도 순서로 정렬된 정보 목록 | 시각 위계(크기·색·위치) 결정 |
| **핵심 액션** | 사용자가 반드시 해야 하는 1~2가지 행동 | CTA 디자인·위치 결정 |

## S1 — 랜딩 페이지 `/`

### 목적
**1초 안에 "이건 연영과/배우용 진단이다"를 인지시키고, 시작 버튼까지 도달시킨다.**

→ 이 화면은 "설명하는 화면"이 아니라 **"이탈을 막는 화면"**. 모든 요소는 시작 CTA 도달을 위해 존재.

### 유저 상황
- **유입 경로 1순위**: 학과 커뮤니티/에타에 누가 글 올림 → 클릭
- **유입 경로 2순위**: 단톡방에서 친구 결과 캡처 보고 링크 받음
- **유입 경로 3순위**: 인스타 스토리에서 캐치프레이즈 보고 클릭
- **디바이스**: 99% 모바일 (세로 한 손)
- **감정·집중도**: 호기심 약, 의심 강 ("이거 풀어볼 만한 거야?"), 이탈 임계점 매우 낮음 (2~3초)
- **시간 압박**: 단톡방 보다가 들른 상태 → 분량이 길어 보이면 즉시 이탈

### 정보 위계 (위→아래 중요도)
1. **도메인 시그널** — "연기인을 위한" 또는 "연영과·배우 전용" (이게 1초 인지의 핵심)
2. **시작 CTA** — "시작하기" 버튼
3. **분량/시간 표시** — "12~14문항 · 약 2~3분" (의심 해소)
4. **서비스명** — "연기 스타일 MBTI" (강력하지만 도메인 시그널 뒤)
5. **서브 카피** — 한 줄 설명 (있으면 좋고)
6. **시각 요소** — 일러스트/16유형 일부 코드 (위트 톤 신호)
7. **푸터** — 만든이/피드백 (가장 약함)

### 핵심 액션
**[시작하기] 탭 → S2 1번 문항**

(보조 액션 없음. 모든 다른 요소는 이 액션을 지원하기 위해서만 존재.)

### 디자인 함의
- 시작 CTA가 **첫 뷰포트(약 640px) 안에** 반드시
- 도메인 시그널은 **CTA보다 위·메인 영역에 굵게**
- 위트 톤이 즉시 느껴지도록 일러스트/코드 시각 요소 1개 (이미지 X, 텍스트 코드 카드 추천)
- 푸터·SNS 링크 등 산만한 요소는 첫 뷰포트 밖
- 로딩 ≤ 2초 (히어로 이미지는 가볍게)

### 이탈 방어 포인트
- "이거 나용 아닌듯" → 도메인 시그널 강화로 방어
- "분량 길어보임" → 2~3분 표시로 방어
- "위트 톤 안 느껴짐" → 텍스트 위트 1개로 신호 (예: 코드 카드에 "MINB - 비 오는 날 뛰쳐나가는 햄릿형" 미리보기)

## S2 — 문항 페이지 `/quiz`

### 목적
**12~14개 시나리오 문항을 마찰 없이 끝까지 풀게 한다.**

→ 풀이 자체가 즐겁기보다는 "어 어느새 다 풀었네" 식의 **무마찰 진행**이 목표.

### 유저 상황
- 시작 직후 (의욕 ★★★)
- 중반 (의욕 ★★, "얼마 남았지?" 욕구 발생)
- 후반 (의욕 ★, "빨리 결과 보고 싶다" 욕구)
- **디바이스**: 모바일 한 손 / 세로 / 단톡방·SNS와 멀티태스킹
- **감정 변화**: 호기심 → 몰입 → "결과 보고 싶음" 으로 가속
- **이탈 임계점**: 1문항이 너무 길거나 톤이 무너지면 즉시 이탈

### 정보 위계 (위→아래 중요도)
1. **선택지 4개** — 사용자가 결정해야 할 본체
2. **질문 카피** — 시나리오 + 질문 (도메인 양념)
3. **진행률** — "3 / 12" + 진행바 (얼마 남았는지)
4. **이전 버튼** — 보조 (있지만 약하게)
5. **서비스명/로고** — 가장 약함 (또는 생략)

### 핵심 액션
**선택지 1개 탭 → 자동으로 다음 문항으로 진행**

보조 액션: ← 이전 (수정 욕구 대응)

### 디자인 함의
- **선택지 카드 = 시각 중심**: 화면 면적의 50% 이상 할애
- 선택지는 명확히 4개로 구분 (카드 사이 간격, 테두리, 호버 효과 없음 = 모바일)
- 진행률은 상단 고정 + 항상 시인 가능 (모달처럼 떠 있지 않게)
- 질문 카피는 모바일 1뷰포트 안에 + 선택지 4개도 같은 화면에 동시 노출 (스크롤 없이)
- 선택 즉시 0.2~0.3초 시각 피드백 후 자동 진행 (탭 1회 = 1선택 = 다음)
- 이전 버튼은 좌상단·작게 (필요한 사람만 발견)
- 광고·산만 요소 일체 금지

### 톤 함의
- 시나리오 카피는 idea-brief의 "진지 30 / 재미 70" 톤 직접 반영
- 선택지 카피도 위트 — 단조로운 "A는 ~다"가 아니라 캐릭터화된 짧은 문장

### 이탈 방어 포인트
- "지루함" → 진행률 가시화 + 선택 즉시 진행
- "선택지가 다 비슷해 보임" → 도메인 양념으로 차별 (리허설/캐스팅/대본 등 구체 단어)
- "끝이 안 보임" → 마지막 문항에서 CTA 변경 ("결과 보기" 강조)

## S3 — 결과 페이지 (본인) `/result/{코드}`

### 목적
**캡처 영역으로 공유 가능성을 극대화하고, 스크롤 영역으로 곱씹기 거리를 제공한다.**

→ 두 가지 목적이 공존하는 유일한 화면. **시각적으로 영역 분리가 명확해야 함.**

### 유저 상황
- 문항 다 푼 직후 (기대감 ★★★)
- **첫 0.5초**: 결과 확인 = 코드와 네이밍 인지
- **다음 2~3초**: "이거 캡처할 만한가" 본능적 판단
- **다음 5~10초**: 캐릭터 묘사 읽음
- **그 이후**: 스크롤 → 역할/장르/천적/베프 곱씹기
- **디바이스**: 모바일 세로
- **감정**: "오 나 이거구나" 또는 "엥 이게 맞아?" (둘 다 공유 충동으로 이어짐)

### 정보 위계 (위→아래 중요도)
**캡처 영역 (1뷰포트 안)**
1. **위트 네이밍** — "비 오는 날 뛰쳐나가는 햄릿형" (캡처 시 가장 먼저 인지)
2. **유형 코드** — `MINB` (정체성 라벨)
3. **한 줄 캐치프레이즈** — SNS 캡션용 위트
4. **캐릭터 묘사** — 3~4줄 (캡처 영역 내에서)

**스크롤 영역**
5. **어울리는 역할/장르** — 차별점, 곱씹기 요소
6. **천적/베프 카드** — 사슬 연장 트리거
7. **공유 액션 그룹** — 이미지/링크/카톡
8. **다시 풀기 버튼** — 가장 약함

### 핵심 액션
1차: **이미지 저장 / 링크 복사 / 카톡 공유** 중 하나 (공유)
2차: **천적/베프 카드 탭** (사슬 연장)
3차: **다시 풀기**

→ 공유 액션 그룹이 가장 강한 시각 위계로 표현.

### 디자인 함의
- **캡처 영역 ↔ 스크롤 영역 시각 경계 필수** — 카드형 컨테이너 / 배경색 변화 / 그림자
- 캡처 영역은 **html2canvas 친화적** 디자인:
  - 그라데이션·복잡한 효과 최소
  - 폰트는 시스템 폰트 또는 웹폰트 빌드 포함
  - 워터마크(서비스명) 우측 하단 자연스럽게
- 위트 네이밍 = **가장 큰 텍스트** (코드보다도 시각 비중 높음)
- 유형 코드 = 강조 색 + 모노스페이스 폰트 (배지 느낌)
- 공유 버튼 3개 = **수평 정렬 + 큰 터치 영역** (한 화면에 모두 보이게)
- 천적/베프 카드 = 클릭 어포던스 명확 (화살표·테두리)
- 다시 풀기는 가장 아래·약한 시각 (실수로 누르지 않게)

### 톤 함의
- 결과 페이지가 idea-brief의 1차 가치(공유·놀이) 압축 — 디자인 완성도가 가장 중요
- 16유형마다 캡처 영역 배경/포인트 컬러 다르게 (수집 욕구·식별성)
- 단, 시각 시스템은 일관 (동일 레이아웃 + 컬러만 변주)

### 이탈 방어 포인트
- "별 거 없네" → 위트 네이밍 임팩트로 방어 (네이밍 자체가 콘텐츠)
- "공유하고 싶지만 어렵네" → 1탭 액션 3개 동시 노출
- "내 결과만 보고 끝" → 스크롤 영역의 천적/베프로 사슬 유도

## S3' — 결과 페이지 (공유 수신자 모드)

### 목적
**친구 결과를 보러 온 사람을 "나도 풀어보기"로 전환시킨다.**

→ S3와 컴포넌트는 같지만 **목적이 다르다**. 본인 결과 표시가 아니라 **CTA 전환**이 1순위.

### 유저 상황
- **유입 경로**: 단톡방/카톡/SNS에서 친구가 보낸 URL 클릭
- **첫 마음**: "친구가 뭐 풀었지" → 호기심 (가벼움)
- **두 번째 마음**: "나도 풀면 뭐 나올까" → 시도 가능성 (이게 핵심)
- **디바이스**: 모바일 세로
- **이탈 임계점**: 친구 결과만 보고 "오케이" 후 닫기 = 가장 큰 위협

### 정보 위계 (위→아래 중요도)
1. **친구 결과 캡처 영역** — S3와 동일 (위트 네이밍·코드·캐치프레이즈·묘사)
2. **"나도 풀어보기" CTA** — 캡처 영역 직후, 첫 스크롤 안 (가장 강한 위계)
3. (선택) 안내 텍스트 — "친구가 풀어본 결과예요" 1줄
4. 그 외 요소는 S3와 동일하지만 시각 위계 약화

### 핵심 액션
**[나도 풀어보기] 탭 → S2 1번 문항**

(S3에는 없던 액션. 이것 하나가 추가되고, 시각 비중 1순위.)

### 디자인 함의
- 모드 판별: localStorage에 본인 코드 저장 여부로 판단
  - 저장 없음 또는 ≠ URL 코드 → **수신자 모드 활성**
- "나도 풀어보기" CTA = 캡처 영역 바로 아래 (스크롤 없이 보이는 위치)
- CTA 시각: S3의 공유 버튼보다도 강한 강조 (primary 컬러 + 풀 폭)
- 안내 텍스트(선택): "친구가 풀어본 결과예요" — 캡처 영역 상단에 작게
- 친구 결과지 자체는 S3와 완전히 동일 (혼선 방지)

### 톤 함의
- "친구의" 결과를 강조하면 거리감 → 결국 "본인도 풀어볼 만한 것"으로 자연 전환되도록
- 안내 톤은 가볍게: "친구가 풀어본 결과예요. 나도 풀어볼까?" 식

### 이탈 방어 포인트 (S3보다 더 중요)
- "친구 결과만 보고 끝" → CTA가 캡처 영역 바로 아래 + 강한 시각
- "분량 의심" → CTA 옆에 "2~3분" 보조 텍스트
- 본인이 한 번 풀었으면 자동으로 S3 모드 (안내 미노출 = 혼선 없음)

## S4 — 404 / 잘못된 코드 페이지

### 목적
**비정상 URL 진입자를 막다른 길에서 구해서 정상 흐름(랜딩 또는 풀이)으로 보낸다.**

→ 발생 빈도는 낮지만 발생 시 100% 이탈 방지가 목표.

### 유저 상황
- **유입 경로**: 오래된 링크 / 오타 / 의도적 URL 변조
- **감정**: 당황 + 약간 흥미 (만약 위트 메시지면 호감으로 전환 가능)
- **디바이스**: 다양

### 정보 위계 (위→아래 중요도)
1. **위트 안내 메시지** — "이 유형은 아직 없는 캐릭터예요" 식
2. **처음으로 CTA** — → S1
3. **풀어보기 CTA** — → S2 (바로 풀고 싶은 사람용)
4. (선택) 일러스트/그래픽 — 위트 톤 강화

### 핵심 액션
**[처음으로] 또는 [풀어보기] 중 하나 탭**

### 디자인 함의
- 무미건조한 "404 Not Found" 금지 → 위트 톤 유지가 더 중요
- CTA 2개 병렬 배치 — primary 강조 1개 + secondary 1개
- 페이지 자체는 매우 가볍게 (콘텐츠 적음, 로딩 빠름)

## 📊 화면 간 비교 매트릭스

| 화면 | 목적 | 1순위 정보 | 1순위 액션 | 시각 무게중심 |
|---|---|---|---|---|
| S1 | 시작 도달 | 도메인 시그널 | 시작하기 탭 | CTA 버튼 |
| S2 | 무마찰 진행 | 선택지 4개 | 선택 1개 탭 | 선택지 카드 |
| S3 | 공유 + 곱씹기 | 위트 네이밍 | 공유 액션 | 캡처 영역 |
| S3' | 전환 (나도 풀기) | 친구 결과 + CTA | 나도 풀어보기 탭 | 강조 CTA |
| S4 | 막다른 길 회피 | 위트 안내 | 처음으로 탭 | CTA 2개 |

## 🎯 Step 2.2로 넘기는 것

각 화면의 4축 분석을 토대로, 다음을 만족하는 **레퍼런스 서비스**를 찾는다:
